## 3. 自律判断の3レベル（最重要）

Claude Code は、以下の3レベルで判断・行動してください。

### 🟢 GREEN：自動で判断・実行（事後報告も不要）

- コード実装の細部（変数名、関数分割、ファイル配置）
- 既存ドキュメントに準拠するライブラリ選定
- バグ修正
- リファクタリング（外部APIに変更がないもの）
- テスト追加
- 開発依存パッケージのバージョン更新（patch/minor）
- スタイル・UI の細部調整
- ドキュメントの誤字修正・補強

**GREEN の作業は黙々と進めて、デイリーログにまとめて記録するだけでOK。**

### 🟡 YELLOW：実行＋事後報告（Supervisor の確認は不要）

- 新機能の追加（ROADMAP.md の Phase 内）
- 主要ライブラリの新規追加
- データモデルの拡張（既存スキーマと互換性あり）
- API エンドポイントの追加
- セキュリティルールの調整
- パフォーマンス改善
- メジャーバージョンアップ（package.json）
- ステージング環境へのデプロイ
- GitHub Actions ワークフローの追加・変更

**YELLOW の作業は、PRに「[YELLOW]」プレフィックスを付けて自動マージ。週次レポートでまとめて報告。**

### 🔴 RED：必ず Supervisor の承認を得る

- **本番環境へのデプロイ**
- **データモデルの破壊的変更**（既存データに影響）
- **価格・課金プランに関わる変更**
- **利用規約・プライバシーポリシーの変更**
- **ブランディングの変更**（ロゴ・カラー・サービス名）
- **新しい外部サービスの導入で月額コストが発生するもの**（月¥1,000以上）
- **第三者APIとの新規連携**
- **採用・解雇の意思決定**
- **法的・契約関連の判断**
- **個人情報の取り扱いを変える**変更

**RED の場合：**
1. PR を作成し、タイトルに「[RED] 要承認」を付ける
2. PR description に「決裁案」を3案以上提示
3. 推奨案を明示し、理由を書く
4. リスク・代替案も明記
5. Supervisor から GitHub のコメントもしくは Cowork チャットで承認をもらう
6. 承認後にマージ・実装

**Supervisor の応答が48時間ない場合、推奨案で進めて構いません（事後報告で）。ただし RED 項目のうち「価格・契約・法的・本番デプロイ」は必ず承認を待つこと。**

## 4. 作業の進め方

### デイリーサイクル

1. **セッション開始時：**
   - `CLAUDE.md`、`AGENT_CHARTER.md`（このファイル）、`ROADMAP.md`、`docs/daily_log.md` の最新を読む
   - 前日のログを確認、今日の TODO を3〜5項目に絞る
2. **作業中：**
   - GREEN なら黙々と実行
   - YELLOW は実装＋PR作成
   - RED は決裁案を作成
3. **セッション終了時：**
   - `docs/daily_log.md` に日付・作業内容・所要時間・コミット一覧を追記
   - 翌日の TODO を3項目書いて締める

### 週次サイクル（毎週金曜）

1. `docs/weekly_reports/YYYY-WW.md` を作成
2. 含める内容：
   - 今週の成果（マージしたPR一覧・対応Issue）
   - 来週の計画（優先順位順に3〜5項目）
   - リスク・ブロッカー
   - Supervisor への質問・確認事項
   - KPI 進捗（Phase1 完了率、コードカバレッジ、エラー率など）
3. リポジトリに push し、Cowork チャットで Supervisor に「週次レポート出ました」と通知

### マイルストーンサイクル（Phase 単位）

1. Phase 開始時：
   - `ROADMAP.md` の該当 Phase の DoD（完了の定義）を確認
   - サブタスクを GitHub Issues に切り出す
   - Project Board でカンバン管理
2. Phase 終了時：
   - DoD すべてのチェック
   - 残課題を次 Phase に持ち越し or 別 Phase 化
   - リリースノート作成
   - Supervisor とのレビュー MTG を提案

## 6. 品質基準（妥協しない領域）

以下は **必ず守る** こと：

- ✅ **TypeScript strict モード**：`any` を使わない
- ✅ **テストカバレッジ**：新規コードは 80% 以上
- ✅ **セキュリティ**：Firestore セキュリティルール必須、認証・認可チェック必須
- ✅ **アクセシビリティ**：WCAG 2.1 Level AA 準拠を目指す
- ✅ **パフォーマンス**：モバイル 3G で初回ロード 3秒以内
- ✅ **モバイル対応**：iPhone Safari、iPad Safari、Android Chrome 必須
- ✅ **エラーハンドリング**：ユーザーに分かりやすいエラーメッセージ
- ✅ **i18n 準備**：将来の英語化を見据えてキー化（最低 ja-JP は揃える）
- ✅ **コミット**：Conventional Commits（`feat:`, `fix:`, `docs:`, `chore:` など）
- ✅ **PR**：1機能=1PR。1,000行を超える場合は分割
- ✅ **ドキュメント**：CLAUDE.md と ARCHITECTURE.md は変更があれば必ず更新

## 7. エスカレーション（早めに上げる）

以下の状況になったら **即座に Supervisor に連絡** してください：

| 状況 | 通知方法 |
|---|---|
| 本番環境でのインシデント・障害 | Cowork チャット + GitHub Issue（緊急ラベル） |
| データ漏洩の可能性 | Cowork チャットで即時連絡 |
| 法的リスクのある事項 | RED として承認待ち |
| 予算超過の懸念 | 週次レポートで早めに警告 |
| RED 案件で48時間応答なし | リマインダー送信 |
| 自分（Claude Code）の能力では判断不能な案件 | 「決裁不能。Supervisor 判断要請」として PR |

## 8. セキュリティ・倫理ガイドライン

- 🔒 **シークレット**：API キー、トークンを絶対にコミットしない（`.env.local` 使用、`.gitignore` 設定）
- 🔒 **個人情報**：選手の名前・写真・身長体重などはチーム内アクセスのみ
- 🔒 **PII の暗号化**：保存時に暗号化
- 🔒 **依存パッケージ**：Snyk または `npm audit` で脆弱性チェック
- 🔒 **コードレビュー**：セキュリティ関連の変更は必ず Supervisor にも確認依頼
- 🚫 **競合製品のコードコピー**：禁止
- 🚫 **JCA / USA Cheer の公式ロゴ・名称**：許諾なく使わない
- ✅ **AI 利用の透明性**：このプロジェクトは AI 開発であることをドキュメントに明記
